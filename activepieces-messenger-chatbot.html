// overview

After shipping the first Messenger AI agent on Make.com, this project rebuilds the same category of system — a Facebook Messenger chatbot with memory, AI-generated replies, and automatic logging — on Activepieces. The goal wasn't just to duplicate the result, it was to get hands-on with how a different automation platform handles the same problems: expression syntax, trigger sample data, step-level testing, and webhook verification all work differently here than they did in Make.com.

The flow catches every incoming Meta webhook event on a single trigger, then routes it two ways: verification handshakes get echoed straight back (required for Meta's initial webhook setup), while real user messages get parsed, checked against existing conversation records to avoid duplicate processing, sent to Groq AI for a reply, logged to Activepieces Tables with a live timestamp, and sent back to the user via the Messenger Send API.

// why rebuild the same thing on a different platform

→ Platform fluency, not just a working bot

  • Make.com and Activepieces solve the same problem with genuinely different mental models
  • Activepieces expressions are step-ID based, not generic aliases — "trigger" isn't always valid
  • Trigger sample data has to be populated deliberately (via a built-in test-request form) before the picker can generate correct references
  • Step-level "Retest Step" debugging surfaced a stale-cache issue that looked like a code bug but wasn't
  • Same core architecture — webhook, parse, dedup, AI call, log, reply — proven portable across builders
  • Part of the planned progression: Make.com → GoHighLevel → Activepieces/n8n → multi-agentic systems

// build story

This build started with a working reference point already in hand — the Make.com Messenger agent — which made it a useful stress test for how much of that architecture (and how much debugging instinct) actually transfers to a new platform.

The starting point

Rebuild a proven architecture on Activepieces

Instead of designing a new bot from scratch, the same core pipeline from the Make.com build — webhook, memory, AI reply, logging — was rebuilt on Activepieces, on a separate Facebook Page used purely for testing.

Hit a wall

The hub.challenge verification wouldn't resolve

The Raw Body field kept returning an empty response even though the trigger data existed. It traced back to how Activepieces stores trigger sample data separately from live test runs — the picker had nothing real to reference until a test request was sent through the builder's own test-data form, not just the live URL.

Debugging instinct tested

A "bug" that turned out to be a stale cache

senderId briefly showed up with a stray space. Rather than assume the parsing code was wrong, the Code step was audited line by line, confirmed as a pure passthrough with no string manipulation — and the space was traced to a step panel that simply hadn't been retested since the sample data was cleaned up.

Wired properly

Real timestamps, not placeholders

lastUpdated started as a static "now" string. It was replaced with a real new Date().toISOString() generated directly inside the existing parsing step, then mapped into the table write through the picker — confirmed with a live ISO timestamp in the logged record.

Where it stands

Fully wired, gated by App Review

The full 12-step pipeline runs cleanly end to end — verification, parsing, dedup, AI reply, logging, and reply delivery. The only thing standing between this and a public bot is Meta App Review, since the connected app is still in Development mode.
