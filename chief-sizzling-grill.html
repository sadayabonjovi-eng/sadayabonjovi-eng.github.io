⚠ This is a practice project, not a live client engagement. Chief Sizzling Grill is a fictional BBQ restaurant built to learn how no-code CRM automation actually behaves end-to-end — not just how to drag boxes together, but why each piece does what it does. No real customers, no real data. Everything below — the bugs, the fixes, the destroy-testing — is real work against a simulated business.

// overview

The scope covered two halves of the same problem: a landing page with a reservation form customers would actually use, and an automation that takes that submission and safely turns it into a CRM record — without duplicate contacts, duplicate bookings, or silent failures.

Six non-obvious bugs surfaced across two build sessions — the first pass on the core Found/Not Found logic, the second on ownership and follow-up accountability — each one the kind that looks fine in a quick demo and quietly breaks in production. All six were root-caused and fixed before this was called done.

// the landing page

Built entirely inside GHL's Funnel/Site editor as a single-scroll structure, with attention paid to visual cohesion rather than stitching together disconnected template sections.

→ Page structure

  • Hero — bold headline, live-fire photography, dual CTA (Book Now / Order Now)
  • Welcome/About — personal framing + chef photo, avoids generic stock branding
  • Signature Dishes — three-image grid
  • Testimonials — placeholder quotes, flagged for real reviews later
  • Map + address — embedded for local discovery
  • Reservation form — First Name, Last Name, Phone, Select Date, Email

The design decision that mattered most wasn't visual — it was which fields were marked required. Destroy-testing (below) showed that without Phone, Email, First Name, and Select Date all mandatory, blank submissions could slip through and create incomplete, unusable contact records.

// three bugs found by breaking it on purpose

Bug 01

A misleading action label

⚠ What happened

  • GHL's "Find opportunity" action can behave two completely different ways depending on config
  • Its on-screen description changes to match — sometimes reads as a create action, sometimes as a pure search
  • Trusted the first description and recommended removing the separate Create step — had to walk that back

→ The lesson

  • Never assume a component's behavior from its label or a single glance at documentation-style text
  • The actual configured state has to be verified directly, every time
Bug 02

"Found" meant "ever had a reservation," not "has an open one"

⚠ What happened

  • No filter fields were set on Find Opportunity — any past opportunity counted as "Found"
  • Would have silently broken repeat customers — a closed-out reservation from last month would block a new card forever

✓ The fix

  • Added match condition: Pipeline Is Reservations
  • Added match condition: Status Is Open
  • Narrowed "Found" to mean specifically an open, active reservation — fixed same-day duplicates and repeat customers in one change
Bug 03

Contact matching doesn't catch typos

⚠ What happened

  • Found/Not Found logic depends on GHL recognizing repeat submissions as the same contact
  • Contact matching keys on email by default — one typo on a second submission silently creates a new contact
  • This bypasses all Found/Not Found logic entirely, since the system sees a different person

✓ The fix

  • Settings → Business Profile → Contact Deduplication Preferences
  • Primary match field: Email · Secondary match field: Phone
  • A typo'd email on a repeat visit now still gets caught by phone match

// closing the silent-failure gap

Originally the "Found" branch only notified staff — the customer who submitted the form got no response at all, which could look like a silent failure from their side. A second notification was added so the customer isn't left to self-diagnose:

"Hi [Name], thanks for reaching out! It looks like you may already have an open reservation with us. Don't worry — one of our staff will personally check this and make sure everything's in order..."

// part 3 — round-robin assignment & escalation ladder

Once the Found/Not Found logic was solid, the automation was extended past "notify staff" into an actual ownership and accountability system — every new reservation lead gets assigned to a person, that person gets reminded, and if nothing happens within 48 hours, it escalates to the business owner directly.

// Both the "Found" (owner-check) and "Not Found" (always assigns) paths converge into this same shared chain

Bug 04

Building the whole workflow twice

⚠ What happened

  • Found and Not Found branches were each built out independently — Task, Email, Wait, Check, Escalation, repeated in full down both sides
  • Any change to timing or the escalation ladder had to be made twice, with no guarantee the copies stayed in sync

→ The distinction

  • Legitimate duplication: messaging genuinely differs (double-booking tone vs. new-lead welcome tone) — this should stay separate
  • Accidental duplication: the escalation ladder mechanics are identical in both branches — this shouldn't be copy-pasted
Bug 05

Ownerless leads with no path to assignment

⚠ What happened

  • The Found branch assumed every existing opportunity already had an owner and skipped straight to notifying "the owner"
  • In practice the pipeline had a mix — manually created, imported, or workflow-created opportunities with no owner at all
  • Round-robin assignment only ran on Not Found — any ownerless "Found" opportunity fell through the crack permanently

✓ The fix

  • Added condition: Opportunity Owner Is Not Empty right after the Found branch
  • Owner set → proceed straight to the shared Task step
  • Owner empty → round-robin Assign to User first, then continue into the same shared step
Bug 06

Standard Builder can't merge branches back together

⚠ What happened

  • Fixing Bug 05 required both ownership paths to converge into the same downstream Task node
  • GHL's Standard Builder doesn't support dragging a connection from one branch into an existing node elsewhere on the canvas
  • Attempting it just detached the connection with no valid target

✓ The fix

  • Switched the workflow to Advanced Builder
  • Used the dedicated Go To action, built exactly for routing a branch to an existing step elsewhere on the canvas
  • Merged both ownership paths into one shared chain, eliminating a third copy of the logic before it was ever built

// known limitations — documented honestly

Named on purpose, not glossed over, as part of the learning process.

  • No capacity limits yet — every valid submission creates a card, with no max-bookings-per-slot check. Planned on GHL's Calendar tool rather than a custom counter, since a naive counter is vulnerable to phantom slots from skipped or duplicate runs
  • Status field integrity is an assumption, not a guarantee — the Found/Not Found logic depends on staff actually moving completed reservations out of "Open." Skip that consistently, and every past customer eventually reads as a false "Found"
  • Gemini AI, Sheets, and Gmail integrations on the broader scenario are still in progress
  • Rapid-fire, millisecond-level duplicate submissions were deprioritized as low-probability, since the form resets and requires full re-entry after each submission
  • The escalation ladder is structurally verified but not yet execution-tested — the builder sub-account can't send live email, so the merge and branching logic were confirmed by tracing the canvas rather than a live test run. Next step: Execution Logs or a temporary tag-based marker to confirm the routed path at runtime

// what this demonstrates

  • Reading past a tool's surface-level label to verify actual configured behavior
  • Proactively stress-testing for edge cases — typos, repeat customers, duplicate submissions — not just the happy path
  • Fixing a problem in the right layer of the system — a workflow symptom traced back to a contact-matching setting
  • Balancing automated logic with human-facing communication, so no one is left confused by silent system behavior
  • Distinguishing legitimate duplication (different messaging for different scenarios) from accidental duplication (identical logic copy-pasted for no reason) — and only merging the latter
  • Recognizing when a tool's Standard mode has hit a real structural limit, and deliberately switching modes rather than working around it with more duplication
  • Honest scoping of what's done, what's deferred, and why
