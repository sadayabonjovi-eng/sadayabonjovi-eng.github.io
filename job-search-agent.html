// overview

Job hunting is repetitive by nature — the same information gets re-typed and re-tailored, posting after posting. Most "auto-apply" tools solve this by scraping platforms like LinkedIn or Indeed and submitting applications with no human involved. That approach is fast, but built on a foundation that doesn't hold — those platforms explicitly prohibit bots and actively ban accounts caught running them.

This agent takes a different path. It sources only from job boards with free, public APIs meant to be queried programmatically, filters listings against a keyword list, and for every genuine new match, uses an LLM to tailor a professional summary and reorder skills to fit that specific posting — never inventing new experience, only reframing what's already true. The tailored resume and cover letter are generated as real .docx files and emailed directly, attached and ready to send.

Runs entirely on free-tier infrastructure — no server to maintain, no recurring cost, and nothing running against a platform's Terms of Service.

// build story

This project didn't start with a fixed architecture. It went through a few real pivots as constraints — cost, platform rules, tool limits — came into focus.

The original plan

Full auto-apply across every platform

The first idea was the obvious one — an agent that finds jobs and submits applications everywhere automatically, LinkedIn and Indeed included. That plan didn't survive first contact with those platforms' Terms of Service, which explicitly prohibit bots and actively ban accounts caught running them.

Scope corrected

Prepare-and-review, not auto-submit

The design shifted to sourcing only from platforms with free, public APIs meant for programmatic use, and stopping short of submission entirely — the agent hands over a link, a tailored resume, and a cover letter, and a human still sends it.

Scheduler decision

Make.com vs. GitHub Actions

Make.com's free tier offered a faster visual build, but its 1,000-operations-per-month ceiling would have been exhausted almost immediately by daily runs. GitHub Actions has no such ceiling and already hosted the code, so it became the scheduler.

LLM decision

Claude API vs. Groq

Claude's writing quality is stronger, but its free allowance is a one-time trial credit, not an ongoing tier. Groq offers a genuinely free, ongoing allowance — the better fit for a tool meant to run indefinitely during an active job search.

Shipped

Live, tested against real listings, running daily

The first production run found a genuine remote administrative-support match, generated a tailored resume and cover letter, and delivered both by email — end to end, unattended, for free.

// qa case study — the html entity bug

Shipping the pipeline wasn't the end of the story. The first real run surfaced a small but visible formatting bug — the kind that's easy to miss in testing but immediately obvious once real data flows through.

⚠ The Bug

  • Job titles pulled from RemoteOK's API displayed with raw HTML entities
  • "Scheduling & Coordinating" rendered as "Scheduling &amp; Coordinating" in the email
  • Only surfaced once a real live listing containing an ampersand was matched

→ Root Cause

  • The job board's API returns titles and descriptions HTML-encoded
  • The fetch layer stripped HTML tags but never decoded HTML entities
  • Encoded text passed untouched through filtering, tailoring, and into the final documents

✓ The Fix

  • Added a shared _clean() helper using Python's built-in html.unescape()
  • Applied to title, company, and description across all three job sources
  • Fixed once at the source, so every downstream step — filtering, tailoring, documents, email — inherits clean text

→ Why It Matters

  • Invisible in code review — only showed up once real API data flowed through the pipeline
  • Root-caused at the point text enters the system, not patched at the point it's displayed
  • The kind of small detail that separates a script that "mostly works" from one that ships clean output every time
